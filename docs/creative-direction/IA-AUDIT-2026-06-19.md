# Information Architecture Audit — Creative Intelligence Layers
## Wings Global Trade · Product Detail Page
**Date:** 2026-06-19 | **Mode:** COMBINED (Audit + Design) | **Standard:** F1 Data Visualization

---

## PREMISE

Every component below was built as a creative endpoint: seeded by specs, aesthetically expressive, but the data relationship is unidirectional and opaque. An F1 display does the opposite — every pixel earns its place by reducing uncertainty for a technical reader. The buyer cannot currently read back from the visuals to the information they encode.

The fix is not redesign. The architecture is sound. The fix is: surface the data that's already in the system.

## COMPONENT DIAGNOSES

### 1. WaveformOverlay — Score: 8/100
**What it does:** HP, torque, and RPM map to the frequency (cycle count) of three sine waves in a 48px strip. The signal is `aria-hidden`. No values are readable from the visual.

**What it should do:** This strip sits between the HP meter and the spec table — the exact position where an engineer expects a power curve. An engine's torque band is the most operationally useful single visualization for machinery procurement. Where does peak torque occur? At what RPM does power fall off? What is the usable operating band for an application at 3,500m altitude?

**F1 standard replacement — `EnginePowerBand`:**
- X-axis: RPM range extracted from specs (e.g., 800–3,600 RPM)
- Two curves: torque (Nm) + power (HP), both normalized to the strip height
- Peak torque RPM labeled with a gold vertical tick + value: "Max torque · 320Nm @ 1,600 RPM"
- Shaded operating band (where torque stays above 80% of peak) — the efficiency zone
- Color encoding: gold = HP curve, warm white = torque curve
- Falls back gracefully when only HP is known (single normalized curve)

This transforms from decoration to the only honest engine performance chart in the LatAm import market.

---

### 2. CellularAutomaton — Score: 2/100
**What it does:** GVW and HP seed a Conway grid. Toroidal wrap. 600ms tick. 32px strip. Zero information utility. User has no mechanism to know it is spec-seeded.

**What it should do:** This 32px strip has exactly the right footprint for a catalog position display — where does this product sit relative to category min/max on each of the 6 key axes?

**F1 standard replacement — `SpecPositionBand`:**
- 6 horizontal range bars stacked in 32px total height (each ~4px)
- Each bar: category min → max, with this product's value as a gold dot
- Labels at left: HP · CARGA · GVW · VELOCIDAD (abbreviated to fit)
- Category ranges from existing `RANGES` constant in spec-normalize.ts — no new data needed
- Static render (no Conway animation) — this is reference data, not atmosphere

Cost: replaces a component that costs GPU cycles with one that earns its pixels.

---

### 3. TradeRouteAnimation — Score: 12/100
**What it does:** Decorative bezier particles from hardcoded ORIGIN (China coast) to DEST (ZOFRATACNA). Weight seeds count and speed. No text labels rendered to canvas. Aria-label only.

**What it should do:** The route IS the data. A B2B buyer evaluating import logistics needs transit time by leg, container type, and cost range. The animation can remain — but the nodes must speak.

**F1 standard upgrade — `LogisticsRouteCard`:**
- Four canvas-rendered node labels (matching ProvenanceRibbon naming: Yantian Port, Manzanillo, ZOFRATACNA, Destino)
- Transit day labels per leg: `→ 22–28d →` (static lookup by source_market)
- Particle speed encodes actual leg duration (short Pacific ≠ long Pacific)
- Bottom annotation strip: Container type inferred from weight (`20ft FCL` / `40ft HQ FCL`), estimated freight tier
- The existing `cif-calculator.ts` freight lookup already contains the data. Wire it.

---

### 4. NoiseField — Score: 3/100
**What it does:** HP seeds density and speed of a vector flow field over the gallery. Purely atmospheric.

**Verdict:** Do not attempt to make this informational — flow fields are not readable data types. Reduce opacity from 0.04–0.08 to 0.02–0.03, making it a near-invisible texture. Then layer a small operational envelope badge on the gallery: altitude range + climate type + typical use terrain. The badge is the information; the noise field is the texture behind it.

### 5. SpecFingerprint — Score: 42/100
**What it does:** 6-axis radar, normalized by RANGES, with ±2% deterministic Molnár deviation per slug. Axis labels show with `showLabels=true`. Gold polygon on navy scaffold.

**What is missing:** Absolute values at vertices (buyer sees shape, not numbers), percentile context (where is this in the catalog?), delta display on variant change, and comparison overlay for the comparison bar.

**F1 standard upgrades — all additive to existing component:**
- Vertex hover/tap: tooltip showing raw value + unit ("120 HP", "8,200 kg GVW")
- Ring labels at right edge: "33%" / "66%" / "MAX CAT." (3 characters each, DM Mono 6px)
- Axis labels show unit in parentheses: "HP" → "HP", "CARGA" → "CARGA (kg)"
- `showPercentile` prop: under each vertex label, "Top 23% catálogo" computed from normalized value
- `comparisonSpecs` prop: renders a ghost polygon (30% opacity, warm white stroke) for the first compared product — makes SpecFingerprint the comparison visualization

### 6. ProvenanceRibbon — Score: 35/100
**What it does:** Four nodes (Origin → Port → ZOFRATACNA → Destination) with shimmer particle. Gold dashed line. Cargo ship midpoint icon.

**What is missing:** Port names are "Puerto" (generic), Destination is "Destino" (default), no timing data per leg, no freight context.

**F1 standard upgrades:**
- Source market → named port lookup: `china` → "Yantian · Guangdong", `india` → "Nhava Sheva", `japan` → "Yokohama"
- Transit day labels between nodes: "→ 28d →" (pre-ship + transit estimate)
- ZOFRATACNA node: add admission status indicator (checkmark or clock icon)
- Ship position shifts based on `daysSinceOrder` prop (if available) or animates as currently
- Node 4 receives `destination` prop from the page (e.g., "Lima", "Arequipa") — ProductDetail already knows this from the buyer's region

### 8. ImportReadinessMeter — Score: 48/100
**What it does:** 5 gold segments fill as buyer progresses (arrive → scroll to specs → select variant → open form → submit). Good behavioral architecture. But: segments are unlabeled.

**What is missing:** The buyer sees 3/5 filled and does not know what step 4 means or what action advances them.

**F1 standard upgrades:**
- Below the meter: 5 short labels in DM Mono 8px: `PRODUCTO · EXPLORADO · VARIANTE · CONSULTA · ENVIADO`
- Active segment label bold/gold, inactive at 35% opacity
- On current incomplete step: a right-aligned action prompt: "→ Selecciona variante"
- Step 5 (completion) triggers a brief gold pulse animation — the submission is the decision moment

### 9. AuthenticationMark — Score: 22/100
**What it does:** HP seeds hexagon rotation, payload seeds inner radius. 12 registration ticks. Slug suffix + category code text. Looks like a compliance stamp but encodes nothing verifiable.

**What is missing:** Real certifications. The geometry is spec-derived but unreadable. A buyer who knows compliance looks at this and sees decoration.

**F1 standard upgrades:**
- Parse specs for known certification signals: Euro II/III/IV/V, Stage II/III, CE mark, INDECOPI ref
- Map found certifications to tick groups (12 ticks = 12 possible standards, 6 relevant per category)
- Illuminated ticks (gold, full opacity) = confirmed certification present in specs
- Grayed ticks = not present or not verified
- Tooltip on hover: lists each tick's standard name
- Inner hexagon area encodes the emission class numerically (size = tier level: Euro V > Euro III)

### 10. BlueprintModeToggle — Score: 15/100
**What it does:** Adds `blueprint-mode` CSS class to `<html>`, changing color scheme (navy bg, mono typography, gold grid overlay). Content is identical between modes.

**This is the highest-leverage item in the audit.** Blueprint mode, as conceived, is the right instinct — a technical pro mode. But currently it only changes the dress, not the information. It should reveal data layers that are hidden in standard mode.

**F1 standard — Blueprint Mode as Data Layer:**

Standard mode (current): polished commercial presentation, 4 priority specs, human copy.

Blueprint mode (target): technical instrument mode.
- Replace description text with full spec table (all rows, no priority filter)
- Show normally-hidden metadata: weight distribution estimate, turning radius, ground clearance
- Reveal HS code inferred from category + Supabase data
- Show ZOFRATACNA duty rate from `duty-rates.ts` (already in the codebase)
- Show dimensional data from TechnicalSilhouette with engineering callout annotations
- Show compliance table: all detected certifications with standard numbers
- Add "Exportar ficha técnica" button (downloads current specs as structured data)
- CellularAutomaton strip replaced by SpecPositionBand in blueprint mode
- WaveformOverlay replaced by EnginePowerBand in blueprint mode

This makes Blueprint Mode a real Pro toggle — the kind that separates Wings from every catalog competitor in LatAm.

### 11. FieldReport — Score: 38/100
**What it does:** Four static category-level text strings. One for each category. Every product in `maquinaria-agricola` gets the same text.

**What is missing:** Product specificity. The altitude performance drop for a 50HP tractor is different from a 120HP tractor. The ZOFRATACNA admission class varies by HS chapter. A buyer trusts product-specific intelligence, not category boilerplate.

**F1 standard upgrade:**
- Move generation to API route with product specs + slug in Claude prompt context
- Prompt instructs: generate 3 specific operational notes using actual spec values (HP, weight, GVW)
- Include: estimated altitude performance drop using HP value (formula: ~3% per 300m above 2000m)
- Tag each note: `[ALTITUD]` `[REGULACIÓN]` `[COMPATIBILIDAD]` `[ZONA FRANCA]`
- Link to specific regulatory codes: "Resolución Directoral MTC 2023-0089" (real anchor text, even if URL TBD)
- Cache in Supabase (same pattern as trade_intelligence column — add `field_report_es TEXT`)

### 12. TechnicalSilhouette — Score: 40/100
**What it does:** Four category SVG engineering drawings. Warm-white monoline on navy. Gold baseline rule. Now contextually placed in the ProductPassport with label "Referencia técnica · Vista lateral."

**What is missing:** Dimensional callouts. An engineering drawing without dimensions is a sketch, not a reference drawing.

**F1 standard upgrades:**
- Add wheelbase dimension arrow from actual `wheelbase_mm` spec (when available)
- Overall length estimate arrow (inferrable from wheelbase + category typical ratio)
- Height callout (cab height for tractor, roof height for bus)
- Fork height for forklift (from `capacidad_de_levante` or payload spec)
- Engineering dimension style: hairline arrows, small perpendicular end ticks, numeric value in DM Mono 8px
- Values from `extractNum()` — already in the spec-normalize pipeline
- If spec not available, dimension line renders as dashed with "—" value (honest absence)

## MULTI-AGENT SYNTHESIS

**IA Architect (this audit):** The system has correct bones. The 6-axis normalization pipeline, the cache-first intelligence pattern, the passport format, the blueprint concept — all architecturally right. The failure is the last 10m: spec data is extracted, normalized, and seeded into visuals, but never surfaced back to the reader as readable numbers. Close that loop and the architecture becomes a genuine expert tool.

**Visual Audit perspective:** The information deficit is visible in the visual rhythm. Dense components (SpecFingerprint, TechnicalSilhouette) sit next to ambient components (NoiseField, CellularAutomaton, WaveformOverlay) with no hierarchy signal telling the buyer which to read and which to feel. F1 visualization establishes a strict hierarchy: primary data (large, labeled, persistent), secondary data (medium, labeled on interaction), ambient signal (small, never labeled). Currently everything is ambient signal weight. Promote the primaries.

**Data visualization principle applied:** Every mark needs a referent. A mark without a referent is decoration. The sine waves in WaveformOverlay, the Conway cells, the flow field particles — none have referents a buyer can read. The SpecFingerprint vertices have referents (the normalized spec values) but they're hidden. Surface the referent and the mark becomes information.

## STRATEGIC ANSWER

**Does this product page win against the top 1% of B2B machinery platforms globally?**

**Current: No — but it's two weeks away.**

The page currently wins on two dimensions that almost no B2B machinery platform achieves: (1) visual seriousness without being sterile, and (2) information layering architecture that is genuinely novel (SpecFingerprint, blueprint mode concept, passport format). This is meaningful — the platform reads like a precision instrument rather than a marketplace.

It loses on one dimension that matters most for conversion: a procurement professional cannot make a purchasing decision faster from this page than from a PDF datasheet. The information they need (altitude performance, ZOFRATACNA duty rate, transit time, container type, real compliance certifications) exists in the system — in `cif-calculator.ts`, `duty-rates.ts`, `spec-normalize.ts`, the Supabase schema — but is not surfaced to the page.

**The five items that close the gap to global top 1%:**

1. **EnginePowerBand** (replace WaveformOverlay) — the single most useful machinery spec visualization that no LatAm competitor provides
2. **SpecFingerprint with values + comparison overlay** — makes the radar chart the fastest cross-product comparison tool on any B2B catalog page
3. **Product-specific FieldReport** — AI-generated at product level, not category level; the honest operational intelligence competitors won't invest in
4. **BlueprintModeToggle as data layer reveal** — the Pro toggle that signals platform depth to technical buyers and reveals HS codes, duty rates, dimensional data
5. **ProductPassport: HS code + duty rate** — the single fact that closes the information gap between "I want this machine" and "I know what it costs to import this machine"

With those five, the page becomes: a precision instrument that presents every piece of information a serious LatAm machinery importer needs to move from product page to inquiry — without leaving the page, without calling a broker, without downloading a PDF.

That is the standard. It is achievable with existing data assets.
